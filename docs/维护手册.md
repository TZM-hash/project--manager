# 项目管理系统维护手册

版本：2026-07-03  
项目：ProjectManager

## 1. 维护目标

本文档用于指导管理员、开发人员和运维人员对 ProjectManager 系统进行日常维护、数据备份、问题排查、版本升级和代码变更管理。

## 2. 日常维护清单

### 每日

1. 确认网站可访问。
2. 确认 SQL Server 服务正常。
3. 检查是否有用户反馈登录、按钮、导入、导出异常。
4. 如有生产日志，检查最近错误日志。
5. 确认数据库备份任务执行成功。

### 每周

1. 抽查项目列表、项目详情、月结、规划中专案、保养订单页面。
2. 抽查 Excel 导出是否正常。
3. 检查数据库大小和磁盘空间。
4. 检查管理员账号和弱密码账号。
5. 清理发布目录外的临时压缩包和日志。

### 每月

1. 月结前备份数据库。
2. 月结后抽查批次明细。
3. 导出一份管理员全量 Excel 备份。
4. 检查项目状态字典是否仍符合业务流程。
5. 检查未结案报表和甘特图导出。

## 3. 数据备份

### 3.1 数据库备份

建议使用 SQL Server Management Studio 或 SQL Agent 定时备份。

手动备份 SQL 示例：

```sql
BACKUP DATABASE ProjectManager
TO DISK = 'D:\Backup\ProjectManager_20260703.bak'
WITH INIT, COMPRESSION;
```

恢复 SQL 示例：

```sql
RESTORE DATABASE ProjectManager
FROM DISK = 'D:\Backup\ProjectManager_20260703.bak'
WITH REPLACE;
```

恢复前必须确认当前数据库是否需要另行备份。

### 3.2 Excel 备份

管理员可在系统中使用“数据导入导出”功能导出全量 Excel。

该备份适合：

- 快速检查业务数据。
- 小规模迁移。
- 给业务人员留档。

不适合完全替代数据库备份，因为数据库还包含 Identity、审计和迁移状态等更完整信息。

## 4. 用户和权限维护

用户管理入口：

```text
后台管理 -> 用户管理
```

维护原则：

1. 新用户创建后分配对应角色。
2. 离职或停用用户建议设置为不启用，不建议直接删除。
3. 管理员账号数量要控制。
4. 默认管理员密码上线后必须修改。
5. 批量导入用户后抽查角色是否正确。

角色说明：

| 角色 | 维护建议 |
| --- | --- |
| Administrator | 只给系统管理员 |
| ProjectStaff | 给需要维护项目进度的人员 |
| Leader | 给需要查看项目全局信息的领导 |
| Viewer | 给只读查询人员 |
| SubCaseContact | 给请购/子案对接相关人员 |

## 5. 项目数据维护

### 5.1 项目新增

方式：

- 页面新增。
- Excel 批量导入。

注意：

- 年度和工号组合不能重复。
- 项目人员必须能匹配系统用户。
- 进度和收款百分比必须在 0 到 100。
- 结案项目需要结案年月。

### 5.2 项目修改

修改后系统会保存操作记录，用于工作留痕。

建议：

- 重要进度说明写清日期、事项、责任人。
- 使用颜色标记时避免过多颜色，保持可读性。
- 金额、状态、项目人员变更后检查详情页审计记录。

### 5.3 项目删除

项目使用软删除。

注意：

- 删除前确认是否已有月结、请购、审计记录。
- 批量删除需要二次确认。
- 删除后默认列表不显示，但历史数据可能仍在数据库中。

## 6. 状态和流程维护

状态设置入口：

```text
后台管理 -> 状态设置
```

默认状态：

1. 已立案
2. 已请购
3. 代码中
4. 试车中
5. 待收款
6. 已结案

维护原则：

- 状态编码 `Code` 作为业务标识，尽量不要随意修改。
- 状态显示名可按业务调整。
- 排序影响时间线和下拉顺序。
- 结案状态应标记 `IsClosed`。
- 停用状态不会作为新选择，但历史项目仍可显示。

## 7. 导入导出维护

### 7.1 支持格式

系统导入统一使用 `.xlsx`。

不要使用：

- `.xls`
- `.xlsm`
- 含宏文件
- CSV 直接改扩展名的文件

### 7.2 导入前检查

1. 下载系统模板。
2. 不要删除表头。
3. 不要修改工作表名称。
4. 日期使用 `yyyy-MM-dd` 或 `yyyy-MM`。
5. 百分比填 0 到 100 的数字。
6. 用户字段优先填系统账号或用户 ID。

### 7.3 用户匹配规则

系统通过 `UserLookupService` 匹配用户，支持：

- 用户 ID。
- 登录账号。
- 邮箱。
- 显示名。
- 部分简繁体中文名转换。
- 逗号、分号、顿号等分隔的多人员。

### 7.4 导入失败处理

如果导入失败：

1. 确认文件扩展名是 `.xlsx`。
2. 确认模板来自当前系统。
3. 删除空行和异常格式行。
4. 检查必填字段。
5. 检查人员是否存在并启用。
6. 分批导入定位错误行。

## 8. 甘特图维护

甘特图入口：

```text
项目详情 -> 甘特图
```

维护规则：

- 整体完成日不能早于整体开始日。
- 细分工作完成日不能早于开始日。
- 当前进度范围为 0 到 100。
- 细分工作名称为空但填写了日期或进度时，系统会按默认名称保存。
- 导出 Excel 会自动按项目跨度调整时间单位。

常见问题：

- 横向太宽：系统会按 2、3、6、12 个月自动合并时间刻度。
- 文字拥挤：减少单个任务名称长度，详细内容放进进度说明。
- 导出没有数据：先保存甘特图，再导出。

## 9. 审计日志维护

审计日志用于工作留痕。

记录内容：

- 操作人。
- 操作时间。
- 操作类型。
- 项目工号。
- 修改摘要。
- 字段前后变化。

维护建议：

- 不要直接删除审计日志。
- 如果数据库增长较大，可后续设计归档表，不建议手动清表。
- 追查项目争议时，优先查看项目详情页操作记录。

## 10. 代码维护流程

推荐流程：

1. 修改前保存 Git 检查点。
2. 先阅读相关 PageModel、Service、Model。
3. 新行为优先补测试。
4. 修改后运行构建和测试。
5. 手动验证关键页面。
6. 再打包或提交。

常用命令：

```powershell
git status --short
.\scripts\dotnet.ps1 build ProjectManager.sln
.\scripts\dotnet.ps1 test ProjectManager.sln
node --check src\ProjectManager.Web\wwwroot\js\site.js
git diff --check
```

## 11. 数据库变更流程

当修改 `Models` 或 `ApplicationDbContext` 时：

1. 新增 EF Core Migration。
2. 检查生成的迁移文件。
3. 更新 SQL 数据字典。
4. 运行测试。
5. 在测试数据库验证升级。
6. 生产部署前备份数据库。

生成迁移示例：

```powershell
.\scripts\dotnet.ps1 ef migrations add MigrationName --project src\ProjectManager.Web\ProjectManager.Web.csproj
```

更新数据库：

```powershell
.\scripts\dotnet.ps1 ef database update --project src\ProjectManager.Web\ProjectManager.Web.csproj
```

## 13. 日志和故障排查

### 13.1 本地命令行运行

运行命令：

```powershell
.\scripts\dotnet.ps1 run --project src\ProjectManager.Web\ProjectManager.Web.csproj --launch-profile http
```

错误会直接输出到控制台。

### 13.2 IIS 运行

检查位置：

- Windows 事件查看器。
- IIS 日志目录。
- ASP.NET Core stdout 日志，如果已启用。

不建议长期启用 stdout 日志，避免日志文件过大。

### 13.3 常见故障

| 问题 | 排查方向 |
| --- | --- |
| 无法打开首页 | 进程是否启动、端口是否占用、防火墙是否放行 |
| 登录失败 | 账号是否存在、密码是否正确、账号是否启用 |
| 跳转后重新登录 | Cookie、域名端口、系统时间、应用池回收 |
| 按钮点击报错 | 查看浏览器控制台、检查对应 handler、运行 `node --check` |
| 导入失败 | 文件格式、模板、必填字段、人员匹配 |
| 数据库报错 | 连接串、SQL 服务、权限、迁移 |
| 甘特图导出异常 | 是否已保存甘特图、日期是否合法 |

## 14. 清理和打包

可清理内容：

- `bin`
- `obj`
- 测试截图临时文件
- 临时日志
- 旧压缩包

不可随意删除：

- `src`
- `tests`
- `docs`
- `scripts`
- `ProjectManager.sln`
- `README.md`
- `Migrations`
- `appsettings.json`

打包给朋友前建议：

1. 运行构建和测试。
2. 删除旧压缩包。
3. 确认没有个人密码或生产连接串。
4. 保留部署手册。
5. 保留 SQL 数据字典。

## 17. 前端资源维护

修改前端时按以下边界处理：

1. 全局变量、应用外壳和 Focus 规则放入 `base.css`。
2. 可复用控件放入 `components.css`。
3. 单一页面或业务模块规则放入 `pages.css`。
4. 主题、字体和动效变体放入 `themes.css`。
5. 新 JavaScript 功能建立独立模块，并在 `site.js` 中按页面选择器载入。
6. 不要重新把大段逻辑放回 `site.js` 或 `site.css`。
7. 修改后需对全部项目 JavaScript 执行 `node --check`，并运行 Release 构建和完整测试。

界面固定文案以繁体中文为准。客户名称、专案名称、厂商名称和人员姓名属于业务资料，不应被界面术语维护直接改写。

## 18. 個人檢視維護與回復

1. 使用者可在支援頁面選擇系統預設，或輸入名稱儲存目前篩選、欄位與行距。
2. 「設為預設」只影響目前使用者與目前頁面；明確網址篩選仍優先套用。
3. 若歷史 JSON 含已移除欄位，服務會忽略未知欄位並保留固定欄，不需直接修改資料庫。
4. 若需回復單一使用者，可刪除該使用者在 `SavedDataViews` 的資料；刪除不影響專案、報表或保養訂單。
5. 回滾 `20260714152636_AddSavedDataViews` 只會刪除個人檢視表，執行前仍應先備份資料庫。
6. 驗證資料工作台時，需測試鍵盤操作、720px 有效寬度（近似 200% 縮放）、表格橫向捲動與狀態色對比。

## 19. 運維監控與背景工作

每日先由管理員開啟「系統管理／運維監控」，確認資料庫、完整備份、磁碟、例外日誌與背景工作服務狀態。

- `/health/live` 失敗：程序未啟動或 HTTP 管線無法回應。
- `/health/ready` 失敗：優先檢查 SQL Server 服務、連線字串、網路及資料庫權限。
- 備份狀態為「未知」：應用程式帳號通常缺少讀取 `msdb.dbo.backupset` 的權限；不代表備份一定失敗，需由 DBA 交叉確認。
- 備份超過 36 小時進入注意，超過 7 天視為異常。
- 磁碟低於 15 GB 或 15% 進入注意，低於 5 GB 或 5% 視為異常。
- Worker 超過 1 分鐘沒有心跳進入注意，超過 5 分鐘視為異常。
- 例外日誌位於 `App_Data/logs/exceptions-yyyyMMdd.jsonl`；工作輸入與輸出位於 `App_Data/operations/input`、`App_Data/operations/output`。

背景工作排查順序：

1. 在「工作進度中心」確認工作狀態、進度、結果摘要與錯誤報告。
2. 若工作長時間停在排隊，檢查運維監控的 Worker 心跳與應用程式程序。
3. 若工作失敗，依工作編號比對應用程式日誌、例外 JSONL 及輸入檔案。
4. 重新啟動後，系統會回復超過十分鐘的遺留執行中工作；先確認原工作是否具重入安全性，再觀察結果報告。
5. 清理 `App_Data/operations` 前必須確認沒有排隊／執行中工作，並保留仍需下載或稽核的輸出。

完整品質門禁：

```powershell
$ErrorActionPreference = 'Stop'
& '.\scripts\quality-gate.ps1'
& '.\scripts\quality-gate.ps1' -RunPlaywright -BaseUrl 'http://127.0.0.1:62383'
```
