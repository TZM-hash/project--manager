# 项目管理系统部署手册

版本：2026-07-03  
项目：ProjectManager

## 1. 部署目标

本文档说明如何在本机、内网服务器或 Windows + IIS 环境部署 ProjectManager 项目管理系统。

默认本地访问地址：

```text
http://localhost:62382
```

默认管理员账号：

```text
用户名：admin
密码：ChangeMe123!
```

首次部署后必须修改默认管理员密码。

## 2. 环境要求

### 2.1 必需软件

| 软件 | 要求 |
| --- | --- |
| Windows | Windows 10/11 或 Windows Server |
| .NET SDK | 9.0 |
| SQL Server | SQL Server Express / Developer / Standard 均可 |
| PowerShell | Windows PowerShell 5+ 或 PowerShell 7 |
| Git | 可选，用于拉取代码 |

### 2.2 数据库要求

默认连接串位于：

```text
src/ProjectManager.Web/appsettings.json
```

默认值：

```json
"DefaultConnection": "Server=localhost\\SQLEXPRESS;Database=ProjectManager;Trusted_Connection=True;TrustServerCertificate=True;MultipleActiveResultSets=true"
```

如果 SQL Server 实例名不同，需要修改 `Server=`。

常见示例：

```text
Server=localhost;Database=ProjectManager;Trusted_Connection=True;TrustServerCertificate=True;MultipleActiveResultSets=true
Server=192.168.1.10;Database=ProjectManager;User Id=sa;Password=你的密码;TrustServerCertificate=True;MultipleActiveResultSets=true
```

## 3. 本地开发部署

### 3.1 进入项目目录

```powershell
cd "D:\AI\project- manager"
```

### 3.2 检查 .NET SDK

```powershell
dotnet --list-sdks
```

需要能看到 9.0 SDK。

### 3.3 还原、构建、测试

```powershell
.\scripts\dotnet.ps1 restore ProjectManager.sln
.\scripts\dotnet.ps1 build ProjectManager.sln
.\scripts\dotnet.ps1 test ProjectManager.sln
```

### 3.4 配置数据库连接

打开：

```text
src/ProjectManager.Web/appsettings.json
```

确认 `DefaultConnection` 指向正确的 SQL Server 实例。

也可直接執行專案根目錄的 `SQL配置工具.exe`。新版工具可測試連線、儲存設定、套用 Migration、檢查健康端點與 `App_Data` 權限，並以 Release 模式啟動系統。儲存設定本身不會自動執行 Migration。

腳本模式：

```powershell
$ErrorActionPreference = 'Stop'
& '.\SQL配置工具.ps1' -Headless -TestConnection
& '.\SQL配置工具.ps1' -Headless -ApplyMigrations
& '.\SQL配置工具.ps1' -Headless -HealthCheck -BaseUrl 'http://127.0.0.1:62383'
```

### 3.5 应用数据库迁移

项目启动时会自动调用 EF Core 迁移。也可以手动执行：

```powershell
.\scripts\dotnet.ps1 ef database update --project src\ProjectManager.Web\ProjectManager.Web.csproj
```

如果提示找不到 `dotnet ef`，先安装工具：

```powershell
dotnet tool install --global dotnet-ef
```

### 3.6 启动系统

```powershell
.\scripts\dotnet.ps1 run --project src\ProjectManager.Web\ProjectManager.Web.csproj --launch-profile http
```

访问：

```text
http://localhost:62383
```

## 4. 生产环境发布部署

### 4.1 发布文件

在项目根目录执行：

```powershell
.\scripts\dotnet.ps1 publish src\ProjectManager.Web\ProjectManager.Web.csproj -c Release -o .\publish
```

发布结果位于：

```text
publish\
```

### 4.2 修改生产配置

建议在生产环境使用 `appsettings.Production.json` 或服务器环境变量覆盖连接串。

示例 `publish\appsettings.Production.json`：

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=你的SQL服务器;Database=ProjectManager;User Id=账号;Password=密码;TrustServerCertificate=True;MultipleActiveResultSets=true"
  },
  "AdminSeed": {
    "UserName": "admin",
    "Email": "admin@example.local",
    "Password": "ChangeMe123!"
  }
}
```

注意：

- 生产环境不要长期保留默认密码。
- 如果使用 SQL 用户名密码，文件访问权限要限制给部署账号。

### 4.3 命令行方式运行

进入发布目录：

```powershell
cd .\publish
```

设置环境：

```powershell
$env:ASPNETCORE_ENVIRONMENT="Production"
$env:ASPNETCORE_URLS="http://0.0.0.0:62382"
```

运行：

```powershell
dotnet ProjectManager.Web.dll
```

内网访问：

```text
http://服务器IP:62382
```

如果其他电脑无法访问，需要检查 Windows 防火墙是否放行端口 62382。

## 5. IIS 部署

### 5.1 安装 Hosting Bundle

服务器需要安装 ASP.NET Core Hosting Bundle，对应 .NET 9。

安装后重启 IIS：

```powershell
iisreset
```

### 5.2 创建 IIS 网站

1. 打开 IIS 管理器。
2. 新建网站，例如 `ProjectManager`。
3. 物理路径指向发布目录 `publish`。
4. 绑定端口，例如 `62382` 或 `80`。
5. 应用程序池选择“无托管代码”。

### 5.3 设置环境变量

在 IIS 网站或应用程序池中设置：

```text
ASPNETCORE_ENVIRONMENT=Production
```

也可以通过 `web.config` 或系统环境变量设置。

### 5.4 权限

IIS 应用程序池身份需要：

- 读取发布目录。
- 访问 SQL Server。
- 如果使用 Windows 身份认证连接数据库，需要给应用程序池身份授权数据库权限。

## 6. SQL Server 准备

### 6.1 创建数据库

如果连接账号有建库权限，EF Core 迁移会自动创建数据库。

如果没有建库权限，请 DBA 先创建：

```sql
CREATE DATABASE ProjectManager;
```

### 6.2 授权

SQL 登录用户至少需要：

- `db_datareader`
- `db_datawriter`
- 初次迁移需要 `db_ddladmin` 或更高权限

迁移完成后，生产运行账号可以收窄权限，但如果系统启动仍自动迁移，就需要保留迁移相关权限。

### 6.3 初始化数据

启动后系统会自动初始化：

- 角色：Administrator、ProjectStaff、Leader、Viewer、SubCaseContact。
- 项目状态：已立案、已请购、代码中、试车中、待收款、已结案。
- 管理员账号：admin。

## 7. 首次上线检查清单

1. `appsettings.json` 或环境变量中的连接串正确。
2. SQL Server 服务已启动。
3. 数据库可连接。
4. `.\scripts\dotnet.ps1 build ProjectManager.sln` 通过。
5. `.\scripts\dotnet.ps1 test ProjectManager.sln` 通过。
6. 网站首页可以访问。
7. 默认管理员可以登录。
8. 登录后立即修改默认密码。
9. 上传 Logo 文件存在于 `wwwroot/img/logo.png`。
10. 导入导出、甘特图导出、打印页面可打开。
11. 防火墙已开放内网访问端口。

## 8. 升级部署

### 8.1 升级前

1. 备份数据库。
2. 备份当前发布目录。
3. 记录当前版本或 Git 提交。
4. 在测试环境执行构建和测试。

### 8.2 升级步骤

```powershell
.\scripts\dotnet.ps1 publish src\ProjectManager.Web\ProjectManager.Web.csproj -c Release -o .\publish-new
```

停止 IIS 网站或运行中的进程。

将 `publish-new` 覆盖到生产发布目录。

启动网站。

检查：

```text
首页
登录
项目列表
项目详情
导入导出
甘特图导出
月结
```

### 8.3 回滚

如果升级失败：

1. 停止网站。
2. 恢复旧发布目录。
3. 如数据库迁移已执行且无法兼容，需要恢复数据库备份。
4. 启动旧版本。

## 9. 常见问题

### 9.1 无法连接 SQL Server

检查：

- SQL Server 服务是否启动。
- 实例名是否正确，例如 `localhost\SQLEXPRESS`。
- 防火墙是否开放 SQL Server 端口。
- 连接串是否启用 `TrustServerCertificate=True`。
- Windows 账号或 SQL 账号是否有权限。

### 9.2 启动时报数据库迁移失败

处理：

1. 查看控制台或 IIS 日志错误。
2. 确认数据库账号有迁移权限。
3. 手动执行：

```powershell
.\scripts\dotnet.ps1 ef database update --project src\ProjectManager.Web\ProjectManager.Web.csproj
```

### 9.3 页面要求重新登录

检查：

- 浏览器是否禁用 Cookie。
- 系统时间是否异常。
- 是否访问了不同域名或不同端口。
- IIS 应用程序池是否频繁回收。

### 9.4 Excel 导入失败

检查：

- 文件必须是 `.xlsx`。
- 不要上传 `.xls` 或带宏文件。
- 是否使用系统下载的模板。
- 用户姓名、账号、ID 是否能匹配系统用户。
- 必填字段是否为空。

### 9.5 端口被占用

查看端口占用：

```powershell
Get-NetTCPConnection -LocalPort 62382
```

停止占用进程：

```powershell
Stop-Process -Id 进程ID
```

或修改 `launchSettings.json` / `ASPNETCORE_URLS` 使用其他端口。

## 12. 健康探測與背景工作部署

部署後需保證應用程式帳號對以下目錄具備讀寫權限：

- `App_Data/logs`：未處理例外 JSONL。
- `App_Data/operations/input`：匯入工作檔案。
- `App_Data/operations/output`：匯出與批量結果檔案。

建議監控端點：

```text
GET /health/live   程序存活；不依賴資料庫
GET /health/ready  服務就緒；需要資料庫可連線
```

以 HTTP 200 為正常；`ready` 非 200 時不要把新流量導向該實例。若要在「運維監控」顯示最近完整備份時間，SQL Server 應用程式帳號需能讀取 `msdb.dbo.backupset`；無此權限時頁面顯示未知，但不影響主要業務功能。

本批新增 Migration：

```text
20260714185744_AddOperationJobs
```

它只建立 `OperationJobs`、兩個查詢索引及到 `AspNetUsers` 的 Restrict 外鍵。升級前仍需完成資料庫與發布目錄備份，再套用 Migration。回滾會刪除 `OperationJobs`，因此必須先保存仍有稽核價值的工作結果與錯誤報告。
